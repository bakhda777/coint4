## Базовые правила
- Всегда разбивать большие задачи на независимые подзадачи.
- Спавнить отдельные агенты, когда подзадачи могут идти независимо.
- Предпочитать read-only exploration перед любыми правками.
- Для поиска по коду, логам и документации использовать дешёвые exploration-агенты.
- Более сильные модели использовать только для реализации, сложного планирования и тяжёлого review.
- Держать каждого агента в узкой зоне ответственности: один вопрос, один write scope, один ожидаемый результат.
- Не запускать параллельно агентов с пересекающимся write scope.
- Если задача упирается в один файл или одну точку решения, делать её последовательно.

## Роли
### Explorer
- Read-only роль.
- Используется для code search, repo exploration, dependency mapping, поиска по логам и docs.
- Не меняет файлы и не принимает финальные решения по правкам.

### Worker
- Реализует изменения в коде, конфиге или документации.
- Держит diff минимальным и не трогает чужие области без необходимости.
- Работает только после того, как exploration собрал факты.

### Reviewer
- Проверяет корректность, регрессии, полноту валидации и недостающие тесты.
- По умолчанию read-only.
- Не переписывает решение без явной причины; сначала формулирует замечания.

### Monitor
- Ждёт долгие операции: тесты, сборки, пайплайны, remote jobs.
- Следит за логами и статусами, затем возвращает короткую сводку.
- Полезен как параллельная роль, пока worker или reviewer делают другую работу.

## Стратегия по моделям
- Spark models: exploration, search, log/doc summarization, monitoring.
- Codex models: implementation и точечные правки.
- Strong reasoning models: planning, architecture trade-offs, complex review.
- Если хватает дешёвой модели, не повышать класс модели без причины.

## Когда запускать агентов параллельно
- Когда подзадачи независимы по файлам, директориям или типу артефактов.
- Когда нужно одновременно исследовать несколько источников: код, docs, логи, конфиги.
- Когда monitor может ждать долгую операцию, пока другие агенты продолжают работу.
- Когда reviewer может проверять уже готовый diff, пока отдельный explorer собирает дополнительный контекст.

## Когда не распараллеливать
- Когда несколько агентов хотят менять один и тот же файл.
- Когда следующая подзадача полностью зависит от результата предыдущей.
- Когда нужно принять одно архитектурное решение до начала реализации.

## Как делить задачи на подзадачи
- Делить по write scope: разные файлы или независимые каталоги.
- Делить по стадии: plan, exploration, implementation, review, monitoring.
- Делить по источнику истины: код, документация, логи, инфраструктурный статус.
- Для каждой подзадачи заранее фиксировать owner, входы, done condition и expected output.
- Если подзадача не формулируется в 1-3 предложениях, её нужно дробить дальше.

## Рекомендуемый workflow
1. `plan`:
   определить цель, ограничения, source of truth и безопасный split на подзадачи.
2. `exploration`:
   запустить one or more Explorer agents для чтения кода, логов, docs и зависимостей.
3. `implementation`:
   запускать Worker agents только по независимым write scopes.
4. `review`:
   дать Reviewer проверить корректность, регрессии и пробелы в валидации.
5. `merge results`:
   объединить выводы, синхронизировать final diff и подготовить единый отчёт.

## Как координировать и сливать результаты
- Orchestrator сначала собирает факты от Explorer agents и устраняет противоречия.
- Изменения от Worker agents сливаются только по disjoint write scopes; конфликтующие правки объединяются последовательно.
- Reviewer подтверждает итоговый diff или возвращает findings до merge.
- После merge нужно один раз прогнать релевантную валидацию и зафиксировать итог.
- Для optimization/runtime задач итоговые заметки писать в `docs/optimization_state.md` и `docs/optimization_runs_YYYYMMDD.md`, если это часть задачи.

## Формат ответа каждого агента
- `files changed`: список путей или `none`.
- `commands executed`: команды, которые агент реально запускал, или `none`.
- `validation results`: тесты, линтеры, dry-run, review findings или `not run`.
- `short summary`: 2-4 строки с главным итогом.

## Практический паттерн
- Сначала 2-3 Explorer agents параллельно собирают факты.
- Затем 1-2 Worker agents делают минимальные правки в независимых областях.
- Потом Reviewer проверяет diff и Monitor дожидается долгих проверок, если они есть.
- Финальный orchestrator объединяет ответы агентов в один результат для пользователя.

Смежные проектные детали и более глубокий optimization-процесс описаны в `docs/agent_playbook.md`.
